## 1. 产品定位

FoodMap V1.0 是一个**本地优先的个人美食手账地图**。用户可以在地图上记录餐厅或美食地点，并为每个地点补充照片、评分、到访时间、文字笔记、标签和图层分类。

V1.0 不做大众榜单，不做社区，不做实时协作，也不追求复杂营销感。它首先是一个用户自己使用的地图记录工具：

> 把吃过、想再去、值得推荐、需要避雷的美食地点，像旅行手账一样钉在地图上。

### 1.1 一句话说明

**FoodMap：一张带照片、评分和回忆的私人美食地图。**

### 1.2 产品关键词

- 地图优先
- 本地记录
- 旅行手账
- 暖色纸感
- 低干扰操作
- 图层分类
- 照片缩略图
- 只读分享
- 导入导出

## 2. 方案四设计方向

方案四采用“旅行手账风”，但必须避免变成杂乱拼贴。设计重点不是堆贴纸，而是让地图工作台拥有温暖、可回忆、轻手账的气质。

### 2.1 视觉目标

| 维度 | 要求 |
| --- | --- |
| 整体感受 | 像一本干净的旅行美食手账，而不是复杂运营页面 |
| 地图表现 | 地图仍是主角，使用低饱和、暖色、浅纹理底图风格 |
| 卡片表现 | 纸张质感、圆角、细描边、轻阴影，不使用厚重玻璃拟态 |
| 装饰元素 | 可少量使用胶带、邮戳、手绘线条、纸角，但只作为氛围点缀 |
| 信息密度 | 比前一版更克制，用户一眼只看当前任务相关信息 |
| 移动端 | 单手可操作，底部抽屉为主，不使用复杂左右栏 |

### 2.2 设计克制规则

1. 每个主界面最多出现 1 个装饰性元素，如纸夹、邮戳、胶带角。
2. 地图上的图钉数量可以多，但卡片和工具按钮必须少。
3. 不在同一屏同时展开图层、筛选、详情、样式面板。
4. 所有编辑入口都隐藏在明确按钮后，不默认铺开表单。
5. 分享页只读，不能出现“编辑”“删除”“新增”等操作。
6. 移动端首屏只保留搜索、地图、定位、图层/筛选入口、底部抽屉。

## 3. V1.0 用户目标

### 3.1 核心用户

1. **美食记录型用户**  
   吃过一家店后，希望用地图、照片和文字记录体验。

2. **旅行规划型用户**  
   去一个城市前后，希望整理想吃清单、已吃店铺和城市美食记忆。

3. **朋友分享型用户**  
   希望把自己整理的一组美食地点以只读方式分享给朋友查看。

4. **本地收藏型用户**  
   想把常去店、想吃店、踩雷店分层管理，而不是散落在地图收藏夹或聊天记录里。

### 3.2 用户要完成的事情

- 我想在地图上新增一个美食地点。
- 我想给这个地点加照片、评分、到访时间和笔记。
- 我想用图层区分“必吃餐厅”“咖啡馆”“甜品饮品”“想去清单”。
- 我想按关键词、城市、评分、标签和到访时间找回记录。
- 我想隐藏某个图层，只看一类地点。
- 我想生成一份本地只读分享快照。
- 我想导出 `.foodmap.json`，在另一个浏览器或设备导入查看。

## 4A. 已完成扩展范围：P8-P14

> 本节是 V1.0 之上的当前阶段扩展。V1.0 的本地优先、无账号、无后端、公网永久分享非目标继续有效。

### 4A.1 历史阶段定位

历史阶段目标是把 FoodMap 从“已通过验收的个人美食地图 + 已核验武汉扫街榜”推进为一个**可长期维护、可真实数据刷新、可被智能体稳定调用的本地优先美食地图工具**。

本阶段不改变产品边界：用户的个人记录仍然存储在浏览器本地；扫街榜仍然是推荐覆盖层，不会自动变成个人记录；任何未核验或存在冲突的 POI 不能成为地图图钉。

### 4A.2 历史阶段用户目标

- 我想在榜单数据发生变化时，看到哪些店新增、下架、改名或迁址，而不是直接污染地图。
- 我想在详情页知道图片、街景或公开证据是否与门店匹配。
- 我想更快录入自己的到访记录，尤其是名称、评分、时间、照片和标签。
- 我想在很多图钉中按商圈、评分、距离、核验状态和个人/推荐来源快速查找。
- 我想让伴随式 Agent 或其他智能体安全地读取、筛选、导出和建议保存地点。
- 我想确认当前版本经过真实数据、截图、自动化测试和 PRD 规格复检。

### 4A.3 历史阶段包含范围

| 模块 | P8-P14 要求 |
| --- | --- |
| 真实数据刷新 | 支持重新刷新高德公开榜单并生成新增、下架、改名、迁址、冲突报告 |
| 冲突治理 | 冲突项默认不上图；高风险 POI 需要人工确认后才能覆盖已核验图钉 |
| 图片与街景证据 | 详情页展示图片来源、匹配状态、失败 fallback；支持证据失效审计 |
| 输入体验升级 | 优化必填优先录入、照片保存前预览、地址/标签快速输入、重复地点提示 |
| 查看筛选升级 | 支持个人/推荐分组、商圈/评分/距离/核验状态筛选、列表与地图联动优化 |
| Agent 调用增强 | 提供更稳定的 Agent Bridge 契约、只读查询、批量导出、结构化错误和审计日志 |
| 性能与包体 | 处理首屏和 bundle 风险；地图、推荐面板、图片尽量懒加载 |
| 产品化验收 | 完成 PRD 复检、真实数据复检、截图证据、最终审计报告和发布说明 |

### 4A.4 历史阶段不包含范围

| 非目标 | 说明 |
| --- | --- |
| 不做后台数据服务 | 刷新脚本和本地数据仍以纯前端/本地项目方式维护 |
| 不自动覆盖高风险 POI | 数据刷新发现冲突时必须停在审计状态 |
| 不让 Agent 直接绕过 UI/Domain 规则 | Agent 写入必须走同一套校验、POI 准入和导出逻辑 |
| 不追求实时全网榜单 | 本阶段目标是稳定核验和可审计刷新，不是实时排名服务 |

## 4B. 已完成扩展范围：P15

P15 不废弃扫街榜，而是把产品主体验调整为“空个人图起步 + 扫街榜参考图层 + 智能打点 + 标签过滤 + 地图海报传播”。P8-P14 的真实数据核验成果继续保留，扫街榜 50 店作为首页可显隐图层存在，默认隐藏，不自动污染个人地图。

### 4B.1 用户历程

1. 用户第一次进入 `#/map` 时看到空个人图，底图仍是武汉地图。
2. 首页“扫街榜”按钮可显示或隐藏 50 个已核验参考图钉。
3. 用户点击地图后直接打开新增模态窗，并带入点击坐标。
4. 用户可手动录入店名、地址、评分、到访时间、照片、笔记和标签。
5. 用户可粘贴简介板文本或网页链接；本地解析基础候选，外部 Agent 可提交结构化候选。
6. 多候选时，FoodMap 结合坐标距离、当前定位、历史城市/标签和来源置信度排序，并展示理由。
7. 用户确认后才保存图钉，不允许静默自动落图。
8. 用户可打结构化标签：吃没吃过、评价、食物种类、自定义标签。
9. 用户可按标签组合过滤个人图钉。
10. 用户可导出单张地图海报 PNG，利于朋友圈传播。

### 4B.2 范围与非目标

| 类型 | 内容 |
| --- | --- |
| 包含 | 空个人图、扫街榜图层显隐、点击新增、文本识别、Agent 候选、候选排序、结构化标签、标签过滤、地图海报 |
| 非目标 | 任意网页爬虫、账号/云同步、自动保存候选、远程地图瓦片截图导出、默认展示扫街榜 |

## 4C. 本阶段扩展范围：P16 真实地点联动闭环

> P16 以“用户真实打点体验是否闭环”为目标，补齐 P15 之后仍存在的关键缺口：真实 POI 多候选召回、当前定位排序、链接/Agent 候选闭环、手机地图 App 跳转、真实地图感海报传播。P16 继续坚持纯前端、本地优先，不引入账号、后端或云同步。

### 4C.1 P16 用户历程

1. 用户进入 `#/map`，看到空个人图、武汉底图和可显隐的扫街榜参考图层。
2. 用户点击地图或点击底部新增，打开新增地点模态窗。
3. 用户输入店名、地址、简介板文本或网页链接。
4. FoodMap 召回多个地点候选；候选可来自本地文本解析、已保存历史、扫街榜参考和外部 Agent 提交的结构化结果。
5. 候选列表显示名称、地址、来源、距离、置信度和排序理由；用户必须手动选择候选，不能静默落图。
6. 若用户允许定位，当前位置只参与候选排序和距离解释；若拒绝定位，则按点击点、历史城市/标签、候选完整度和来源置信度降级排序。
7. 用户确认候选并保存后，地点进入个人图层，支持结构化标签、照片、评分、到访时间和笔记。
8. 用户在详情页点击“打开地图/导航”，可跳转到手机端地图软件或网页地图。
9. 用户按标签、来源、评分、区域等筛选当前个人图钉。
10. 用户导出一张更接近真实地图查看体验的传播图，包含当前筛选结果、地图感背景、图钉和标题说明。

### 4C.2 P16 包含范围

| 模块 | P16 要求 |
| --- | --- |
| 真实地点候选 | 支持同名多门店候选展示；候选至少包含名称、地址、坐标、来源、置信度和理由 |
| 当前定位 | 支持浏览器 Geolocation 授权、拒绝和失败 fallback；定位只用于排序和辅助展示 |
| 候选排序 | 综合当前位置、地图点击点、历史城市/标签、扫街榜参考、Agent 置信度和字段完整度 |
| 链接/Agent 闭环 | 网页链接不内置任意爬虫；Agent 可提交结构化候选，FoodMap 本地排序和确认 |
| 外部地图联动 | 个人地点和可上图推荐地点详情提供外部地图跳转入口 |
| 地图 App 跳转 | 移动端优先生成高德/Apple Maps/通用 geo 或网页地图链接，失败时复制地址/坐标 |
| 真实地图感海报 | 导出传播图应包含当前筛选图钉、地图感背景、标题、标签摘要和生成时间 |
| 移动端体验 | 390x844 下可完成搜索候选、定位排序、保存、打开地图和导出入口操作 |

### 4C.3 P16 非目标

| 非目标 | 说明 |
| --- | --- |
| 不做后端 POI 服务 | 不引入自建地点搜索后端；真实 POI 搜索优先通过前端 provider、公开链接或 Agent 辅助 |
| 不做任意网页爬虫 | 普通网页链接只作为输入线索；复杂解析由外部 Agent 以结构化候选形式回填 |
| 不自动保存候选 | 任何候选必须用户确认后才能成为个人地点 |
| 不强依赖定位授权 | 用户拒绝定位时仍可用点击点、历史记录和文本置信度排序 |
| 不承诺所有地图 App 都能拉起 | 以高德/Apple Maps/通用 URI/网页地图多策略降级，验收以可点击和 fallback 为准 |
| 不改变扫街榜定位 | 扫街榜仍是参考图层，不默认污染个人地图 |

### 4C.4 P16 出门体验标准

- 用户能在移动端从空个人图完成一次“输入店名或链接 -> 多候选选择 -> 保存 -> 打开外部地图 -> 标签过滤 -> 导出传播图”的完整链路。
- 候选排序必须可解释，且定位拒绝场景不能中断流程。
- 外部地图跳转必须出现在地点详情中，而不是隐藏在调试或 Agent 接口里。
- 传播图不能是空白图，不能导出与当前筛选结果明显不一致的地点集合。
- 扫街榜 50 条真实核验基线仍需通过。
- P16 实施细则、候选字段、provider 顺序、排序理由、Agent action 和验收 fixture 以 [P16 Real Place Linking Implementation Contract](./p16-real-place-linking-implementation-contract.md) 为准；如果实施契约无法支撑本节体验目标，必须先修订契约再开发。

## 4D. 本阶段扩展范围：P17 体验可信度与移动主路径收束

> P17 以“真实日常使用是否顺手、可信、可纠错”为目标。P16 已经建立真实地点联动闭环，P17 不改变纯前端、本地优先边界，而是收束用户反复反馈的体验问题：待确认地点难处理、详情页信息层级混乱、移动端查看/挪动/跳转不稳定、筛选栏和图钉视觉不够清楚、分享图还不够适合传播、真实数据量下地图缩放仍需性能验收。

### 4D.1 P17 用户历程

1. 用户进入 `#/map`，首页筛选栏只展示高频开关：扫街榜、参考图层、待确认、标签和更多筛选，所有控件在桌面、平板、移动端都完整可见。
2. 用户看到待确认数量后，可进入“待确认工作台”，逐个处理不确定地点。
3. 待确认工作台展示原始输入、当前坐标风险、候选门店列表、证据说明、确认、跳过和手动挪动入口。
4. 用户点击地点详情时，详情页优先展示地点状态、可编辑标签、核心操作和坐标可信度，再展示照片、评分、时间、地址、笔记、候选校准和外部地图入口。
5. 用户可在详情页始终启动“手动挪动图钉”，移动端和窄屏下也能明确进入、确认、取消和保存。
6. 图钉视觉系统稳定区分已核验收藏、待确认收藏、选中态、扫街榜、参考图层和高密度小点状态，点击和缩放时不闪烁。
7. 用户按标签、状态、来源、评分过滤时，地图、列表、详情入口和分享图使用同一筛选状态。
8. 用户导出分享图时，可预览当前视野/当前筛选结果，生成适合朋友圈传播的单张图。
9. 移动端用户可完整完成：打开地图 -> 筛选待确认 -> 进入详情 -> 修改标签 -> 挪动图钉 -> 打开地图 App -> 导出分享图。
10. Agent 或其他智能体可读取待确认队列、候选上下文和当前筛选状态，但不能直接绕过用户确认固化坐标。

### 4D.2 P17 包含范围

| 模块 | P17 要求 |
| --- | --- |
| 待确认工作台 | 聚合待确认/高风险个人地点，支持候选查看、确认、跳过、手动挪动和处理进度 |
| 详情页信息架构 | 重排详情内容：状态/标签优先、核心操作明确、照片/记录/校准/导航分区，桌面和移动端分别优化 |
| 手动挪动图钉 | 详情页始终可进入挪动模式；移动端有清晰确认/取消条；保存后记录校准状态和审计信息 |
| 首页筛选收束 | 高频开关常驻，低频操作进入更多；所有尺寸下不得截断、遮挡或持久覆盖主任务 |
| 图钉视觉系统 | 建立统一状态色、选中态、缩放态和高密度态；不使用闪烁、误导阴影或难读数字 |
| 分享图导出 | 支持当前筛选/当前视野预览，包含标题、地点数、标签摘要、生成时间和地图感背景 |
| 移动端主路径 | 390x844、430x932 和小平板场景可完成查看、编辑标签、挪动、导航和分享 |
| 性能与真实数据 | 扫街榜、参考图层、个人收藏和待确认数据同时存在时，缩放/筛选/详情打开不卡顿到不可用 |
| Agent 调用边界 | Agent 可读待确认队列和候选上下文；固化坐标、删除、保存仍需 UI/Domain 校验 |

### 4D.3 P17 非目标

| 非目标 | 说明 |
| --- | --- |
| 不做后端 POI 校准服务 | P17 继续纯前端；真实搜索可用浏览器可配置 provider 或 Agent 结构化候选 |
| 不承诺自动修正所有错点 | 用户确认、候选证据和手动挪动仍是可信闭环核心 |
| 不做账号/云同步/多人协作 | 个人数据继续保存在浏览器本地 |
| 不做公开社交产品 | 分享图和本地快照用于个人传播，不建立社区流 |
| 不牺牲地图主视图 | 工作台和详情页不能把首屏变成复杂后台管理台 |

### 4D.4 P17 出门体验标准

- 用户能在桌面和移动端看到待确认入口，并完成至少 3 个待确认地点的“候选确认/手动挪动/跳过”处理。
- 详情页在 1280x900、1440x900、390x844、430x932 下无文本横向截断、无不可滚动区域、无地图跟随滚动导致空白。
- 标签可在详情页新增、删除、持久化；评分标签不再污染普通标签，百分制评分按 `(数字评分 - 65) / 35 * 5` 归一到五分制展示并保留原始分。
- 首页筛选栏在 2048x768、1440x900、1280x820、768x900、390x844 下所有可见控件不越界、不半截显示。
- 图钉选中态仅通过稳定颜色/描边区分；点击、缩放和详情打开时不出现闪烁。
- 分享图导出前有预览，导出结果与当前筛选和图钉可见状态一致。
- 扫街榜 50 条、钉图易参考层、个人收藏和待确认队列均作为真实数据参与验收。
- P17 实施细则以 [P17 UX Trust Implementation Contract](./p17-ux-trust-implementation-contract.md) 为准。

## 4E. 本阶段扩展范围：P18 地点候选搜索与可信校准体验增强

> P18 以“用户能更容易把不准的点修到可信位置”为目标。P17 已经完成待确认工作台、详情页信息架构、移动主路径、筛选栏、图钉视觉和分享图验收；P18 不改变纯前端、本地优先边界，而是围绕地点候选获取、候选确认、手动挪动、分享图传播质量、移动详情精简和 Agent 辅助找候选继续优化。

### 4E.1 P18 用户历程

1. 用户进入 `#/map`，看到个人图、扫街榜参考层、钉图易参考层、待确认数量和当前筛选状态。
2. 用户打开一个待确认或高风险地点，详情页优先展示“为什么不可信”和“可以怎么修”。
3. 用户点击“搜索候选”，系统在纯前端范围内优先使用本地历史、扫街榜、钉图易参考、可配置高德 Web 服务 Key、地图网页链接和 Agent 结构化候选。
4. 候选列表展示名称、地址、来源、置信度、坐标精度、风险理由和证据入口；用户可以确认、跳过、复制搜索词、打开外部地图搜索或改用手动挪动。
5. 用户拖动图钉时，界面展示当前位置、新旧坐标、是否来自候选、是否手动校准，并在保存前明确确认。
6. 用户在移动端详情页看到更精简的首屏：状态、店名、标签、核心动作和校准入口优先；照片、长笔记、候选历史和高级搜索按需展开。
7. 用户使用首页筛选时，系统展示当前筛选摘要和一键清空，让用户知道“为什么地图上只剩这些点”。
8. 用户导出分享图时，可编辑标题，选择“当前视野”或“当前筛选结果”，并看到地点数、标签摘要和生成时间。
9. Agent 可以读取待确认队列、生成候选建议、提交候选和解释风险，但不能直接固化坐标、删除地点或隐藏风险。

### 4E.2 P18 包含范围

| 模块 | P18 要求 |
| --- | --- |
| 候选搜索入口 | 待确认详情和工作台内提供搜索更多候选、复制搜索词、打开外部地图搜索、Agent 提交候选入口 |
| 候选证据模型 | 候选必须显示来源、地址、坐标精度、置信度、风险理由和证据链接/截图/说明 |
| 高德 Web 服务接入边界 | 保持纯前端；Key 只保存在本机浏览器；无 Key 时提供复制搜索词和外部地图网页 fallback |
| 手动挪动升级 | 挪动时展示新旧坐标、确认/取消、校准状态、审计备注，避免误触保存 |
| 移动详情精简 | 首屏聚焦状态、标签、核心动作和校准；照片、笔记、候选历史和高级搜索按需展开 |
| 首页筛选摘要 | 展示当前筛选原因、可见点数、待确认数、扫街榜/参考层状态，并提供一键清空 |
| 分享图升级 | 支持标题编辑、当前视野/当前筛选结果模式、地点数、标签摘要、生成时间和传播友好排版 |
| 个人数据体检 | 对导入或历史个人收藏生成可信度摘要，按已核验、待确认、高风险、手动校准分组 |
| 性能压测 | 在 500、1000、3000 点模拟数据下验收缩放、筛选、列表和详情打开 |
| Agent 边界 | Agent 只能辅助找候选、解释风险和设置筛选；不能绕过用户确认改坐标 |

### 4E.3 P18 开发目标编排

P18 不只是一条“候选搜索”开发线。当前阶段必须按以下工作流并行编排，并在总验收时逐项回归：

| 工作流 | 目标 | 关键输出 | 主要验收 |
| --- | --- | --- | --- |
| W18-A 坐标准确性与候选校准 | 让不可信地点有可继续查找、确认、跳过、手动挪动的完整闭环 | 候选搜索入口、provider fallback、候选证据卡、挪动审计预览 | 10+ 待确认地点端到端处理，确认前不改坐标，刷新后状态保留 |
| W18-B 详情页和移动主路径 | 解决详情页拥挤、截断、滚动和核心动作不可达问题 | 详情页渐进披露、移动底部操作、标签/评分/校准入口重排 | 390x844、430x932、768x900 截图和 E2E 通过 |
| W18-C 首页筛选与图层解释 | 让用户理解当前显示了哪些点、为什么点变少、如何恢复 | 筛选摘要、图层状态、待确认/高风险计数、清空入口 | 多条件筛选后地图、列表、详情、分享图点数一致 |
| W18-D 分享传播体验 | 从“能导出”升级为“可编辑、可预览、可传播” | 分享图标题编辑、当前视野/当前筛选模式、传播排版 | PNG 非空，标题、模式、地点数、标签摘要与当前事实一致 |
| W18-E 数据健康与性能 | 让真实个人收藏、参考层和大数据量不会拖垮主路径 | 数据体检、可信度分组、500/1000/3000 点性能 smoke | 真实数据 + 模拟数据均有截图或 JSON 证据 |
| W18-F Agent 和验收治理 | 让智能体能辅助但不能污染地点事实 | Agent 只读/候选提交边界、negative tests、最终验收报告 | 直接固化坐标/删除/隐藏待确认必须被拒绝 |

以上工作流允许并行设计和开发，但每个工作流进入实现前都必须有对应验收标准；任一工作流出现 blocker，P18 不能整体出门。

### 4E.4 P18 非目标

| 非目标 | 说明 |
| --- | --- |
| 不做后端 POI 服务 | P18 仍不引入自建后端，外部地图能力通过浏览器可配置 provider、网页链接或 Agent 候选接入 |
| 不承诺全自动校准所有地点 | 候选只是辅助，最终坐标固化必须由用户确认 |
| 不存储第三方 Key 到云端 | 高德 Web 服务 Key 仅本地保存，不进入导出分享包 |
| 不做账号、云同步、多人协作 | 个人数据仍保存在浏览器 IndexedDB |
| 不牺牲地图主体验 | 搜索、工作台、候选和分享都必须围绕地图，不变成后台表格工具 |

### 4E.5 P18 出门体验标准

- 用户能从待确认详情发起候选搜索，并在无 provider Key 时获得可用 fallback。
- 至少 10 个真实或确定性待确认地点完成候选搜索/确认/跳过/手动挪动路径验收。
- 候选确认不能静默保存；保存前必须展示来源、坐标精度和风险理由。
- 移动端 390x844、430x932 下详情首屏不拥挤，核心动作和校准入口可见。
- 首页筛选摘要能解释当前可见点数和启用的筛选条件。
- 分享图支持标题编辑和模式选择，导出结果与当前视野或当前筛选一致。
- 500、1000、3000 点模拟数据性能 smoke 不得出现不可用级卡顿。
- Agent 直接固化坐标、删除地点、隐藏待确认状态必须被拒绝。
- P18 实施细则以 [P18 Candidate Search And Trust Calibration Contract](./p18-candidate-search-trust-contract.md) 为准。

## 6. 全局体验原则

### 6.1 地图第一

首屏就是地图，不做营销落地页。所有面板都浮在地图上，但不遮挡主要地图区域。用户进入后应立刻看到：

- 当前城市或默认地图区域
- 已记录的美食图钉
- 搜索入口
- 图层入口
- 新增地点入口

### 6.2 信息按需出现

V1.0 不在首页展示所有字段。默认首页只展示：

- 地图点位
- 图层数量
- 搜索
- 筛选入口
- 当前选中点位的摘要卡片

完整内容只在详情抽屉或详情页中出现。

### 6.3 编辑动作要轻

新增和编辑都必须能快速完成。必填字段最少化：

- 名称
- 经纬度
- 图层
- 评分
- 到访时间

地址、城市、标签、笔记、照片均可后补。

### 6.4 分享页必须可信

分享页要让用户明确知道：

- 这是本地只读快照。
- 不会修改原始记录。
- 不能公开永久访问。
- 如需跨设备查看，需要导入 `.foodmap.json`。

### 6.5 旅行手账风不等于低效率

视觉可以温暖，但交互必须清晰。按钮、输入框、状态反馈要像工具产品一样直接。

## 11. 用户流程

### 11.1 新增点位：地图点击

```text
进入 #/map
→ 点击地图空白处
→ 直接打开新增地点表单，并带入点击坐标
→ 填写名称、评分、到访时间、图层
→ 可选上传照片、添加标签和笔记
→ 保存
→ 地图出现新图钉
→ 详情抽屉打开
```

### 11.2 新增点位：搜索结果

```text
输入店名/商圈/菜系
→ 展示搜索结果列表
→ 点击结果
→ 地图定位到结果
→ 点击“保存为美食点位”
→ 打开新增表单并预填名称、地址、城市、经纬度
→ 保存
```

### 11.3 查看详情

```text
点击地图图钉
→ 地图聚焦该点
→ 桌面端右侧详情抽屉打开
→ 移动端底部详情抽屉半屏展开
→ 用户查看照片、评分、到访时间、地址、标签、笔记
```

### 11.4 编辑点位

```text
打开点位详情
→ 点击“编辑”
→ 打开编辑表单
→ 修改字段
→ 保存
→ 关闭表单
→ 详情和地图图钉同步更新
```

### 11.5 删除点位

```text
打开点位详情
→ 点击“删除”
→ 出现确认弹窗
→ 确认删除
→ 点位从地图和列表消失
→ 显示 toast：已删除地点
```

### 11.6 筛选地图

```text
点击筛选
→ 选择城市、标签、评分、到访时间
→ 点击“查看结果”
→ 地图只显示符合条件的点位
→ 顶部显示筛选条件数量
→ 点击“清空”恢复全部
```

### 11.7 生成分享快照

```text
点击“分享”
→ 输入快照标题
→ 确认生成
→ 系统生成本地快照
→ 显示分享页入口
→ 用户可打开只读页或导出 .foodmap.json
```

### 11.8 导出导入

```text
点击“导入/导出”
→ 选择导出当前地图或当前快照
→ 下载 .foodmap.json

clean profile 中打开应用
→ 点击导入
→ 选择 .foodmap.json
→ 校验通过
→ 生成本地分享快照
→ 打开只读分享页
```

## 12. 状态设计

### 12.1 空状态

#### 首次无记录

标题：

```text
还没有美食图钉
```

说明：

```text
点击地图任意位置，或搜索一家店，开始记录你的第一份味觉记忆。
```

按钮：

```text
搜索地点
在地图上新增
导入 .foodmap.json
```

#### 筛选无结果

```text
没有找到符合条件的地点。
试试放宽评分、标签或到访时间范围。
```

按钮：

```text
清空筛选
```

#### 分享快照不存在

```text
没有找到这个本地分享快照。
如果你是从其他设备打开，请先导入对应的 .foodmap.json 文件。
```

按钮：

```text
导入文件
返回我的地图
```

### 12.2 加载状态

- 地图加载：显示纸感底色 + “正在展开地图...”
- 数据加载：显示小型 loading，不遮挡地图
- 照片处理：缩略图位置显示“正在生成缩略图”
- 导入处理：显示“正在校验文件”

### 12.3 错误状态

| 场景 | 文案 |
| --- | --- |
| 地图 provider 失败 | 地图暂时无法加载，请检查网络或稍后重试。 |
| AMap Key 不可用 | 当前使用备用地图模式。 |
| 照片过大 | 这张照片过大，已尝试生成缩略图；如失败请更换图片。 |
| 导入格式错误 | 这个文件不是有效的 FoodMap 数据包。 |
| 存储失败 | 保存失败，可能是浏览器存储空间不足。 |

### 13.1 色彩

| Token | 建议色值 | 用途 |
| --- | --- | --- |
| `--color-paper` | `#F6EBD6` | 页面纸张底色 |
| `--color-paper-light` | `#FFF8EA` | 卡片底色 |
| `--color-ink` | `#3B2B1F` | 主文字 |

| `--color-line` | `#D8C5A5` | 描边 |
| `--color-primary` | `#C76A32` | 主按钮、重点图钉 |
| `--color-olive` | `#6F7F47` | 成功、自然、筛选确认 |
| `--color-gold` | `#D99A2B` | 星级评分 |
| `--color-danger` | `#B94A3A` | 删除、危险操作 |

### 13.2 字体

中文优先：

- 标题：PingFang SC Semibold / 思源黑体 Bold
- 正文：PingFang SC Regular / 思源黑体 Regular
- 数字：Inter / SF Pro

不建议把所有文字做成手写体。手写感只用于小标签、装饰标注或空状态插画，避免影响可读性。

### 13.3 卡片

- 圆角：16-22px
- 边框：1px 暖灰描边
- 阴影：轻阴影，避免厚重
- 背景：米白或浅牛皮纸色
- 内容：标题、信息、按钮之间留足呼吸感

### 13.4 图钉

| 图层类型 | 图标 | 建议色 |
| --- | --- | --- |
| 必吃餐厅 | star | 橙色 |
| 咖啡馆 | coffee | 咖啡棕 |
| 小吃快餐 | bowl | 橄榄绿 |
| 甜品饮品 | heart | 蜜桃粉 |
| 朋友聚餐 | pin / group | 蓝灰 |
| 想去清单 | pin | 紫灰 |

图钉必须清晰，不使用复杂插画作为 marker。

## 16. UX 验收标准

### 16.1 桌面端

- 首屏可明确识别地图是主角。
- 左侧图层、顶部搜索、右侧详情不会互相遮挡。
- 点位详情打开后，地图仍可见。
- 新增、编辑、删除入口清晰。
- 筛选状态可见，清空筛选入口明确。
- 分享页不出现任何编辑按钮。

### 16.2 移动端

- 地图默认占据主要屏幕空间。
- 搜索栏不遮挡关键图钉。
- 底部抽屉可折叠，不永久遮住地图。
- 图层和筛选入口易触达。
- 新增流程不需要复杂跳转。
- 详情内容可滚动，按钮不重叠。

### 16.3 数据与安全感

- 刷新后数据不丢失。
- 删除前必须确认。
- 导入失败不能破坏当前数据。
- 分享页明确“本地只读”。
- 导出文件名清楚，例如：`foodmap-shanghai-cafe-2026.foodmap.json`。

## 17. V1.0 体验结论

FoodMap V1.0 采用方案四“旅行手账风”后，产品重点应从“功能很多的地图工具”转为“温暖、克制、可持续记录的个人美食地图”。

这一版的成败不取决于装饰是否足够像手账，而取决于三件事：

1. **地图是否始终是主角。**
2. **添加和查看一个美食点位是否足够顺。**
3. **本地分享和导入导出是否让用户觉得清楚、可信、安全。**

V1.0 可以先把图层、标签、评分、照片、到访时间、笔记和本地分享做扎实。复杂自定义字段、多人协作、云端同步、公开地图链接和推荐算法都应留到后续版本。
