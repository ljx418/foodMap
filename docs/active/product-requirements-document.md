## 1. 产品定位

FoodMap V1.0 是一个**本地优先的个人美食手账地图**。用户可以在地图上记录餐厅或美食地点，并为每个地点补充照片、评分、到访时间、文字笔记、标签和图层分类。

V1.0 不做大众榜单，不做社区，不做实时协作，也不追求复杂营销感。它首先是一个用户自己使用的地图记录工具：

> 把吃过、想再去、值得推荐、需要避雷的美食地点，像旅行手账一样钉在地图上。

### 1.1 一句话说明

**FoodMap：一张带照片、评分和回忆的私人美食地图。**

### 1.2 产品关键词

- 地图优先
- 本地记录
- 旅行手账
- 暖色纸感
- 低干扰操作
- 图层分类
- 照片缩略图
- 只读分享
- 导入导出

## 2. 方案四设计方向

方案四采用“旅行手账风”，但必须避免变成杂乱拼贴。设计重点不是堆贴纸，而是让地图工作台拥有温暖、可回忆、轻手账的气质。

### 2.1 视觉目标

| 维度 | 要求 |
| --- | --- |
| 整体感受 | 像一本干净的旅行美食手账，而不是复杂运营页面 |
| 地图表现 | 地图仍是主角，使用低饱和、暖色、浅纹理底图风格 |
| 卡片表现 | 纸张质感、圆角、细描边、轻阴影，不使用厚重玻璃拟态 |
| 装饰元素 | 可少量使用胶带、邮戳、手绘线条、纸角，但只作为氛围点缀 |
| 信息密度 | 比前一版更克制，用户一眼只看当前任务相关信息 |
| 移动端 | 单手可操作，底部抽屉为主，不使用复杂左右栏 |

### 2.2 设计克制规则

1. 每个主界面最多出现 1 个装饰性元素，如纸夹、邮戳、胶带角。
2. 地图上的图钉数量可以多，但卡片和工具按钮必须少。
3. 不在同一屏同时展开图层、筛选、详情、样式面板。
4. 所有编辑入口都隐藏在明确按钮后，不默认铺开表单。
5. 分享页只读，不能出现“编辑”“删除”“新增”等操作。
6. 移动端首屏只保留搜索、地图、定位、图层/筛选入口、底部抽屉。

## 3. V1.0 用户目标

### 3.1 核心用户

1. **美食记录型用户**  
   吃过一家店后，希望用地图、照片和文字记录体验。

2. **旅行规划型用户**  
   去一个城市前后，希望整理想吃清单、已吃店铺和城市美食记忆。

3. **朋友分享型用户**  
   希望把自己整理的一组美食地点以只读方式分享给朋友查看。

4. **本地收藏型用户**  
   想把常去店、想吃店、踩雷店分层管理，而不是散落在地图收藏夹或聊天记录里。

### 3.2 用户要完成的事情

- 我想在地图上新增一个美食地点。
- 我想给这个地点加照片、评分、到访时间和笔记。
- 我想用图层区分“必吃餐厅”“咖啡馆”“甜品饮品”“想去清单”。
- 我想按关键词、城市、评分、标签和到访时间找回记录。
- 我想隐藏某个图层，只看一类地点。
- 我想生成一份本地只读分享快照。
- 我想导出 `.foodmap.json`，在另一个浏览器或设备导入查看。

## 4A. 已完成扩展范围：P8-P14

> 本节是 V1.0 之上的当前阶段扩展。V1.0 的本地优先、无账号、无后端、无云同步、无多人协作、无公网永久分享链接边界继续有效，且不作为后续路线候选。

### 4A.1 历史阶段定位

历史阶段目标是把 FoodMap 从“已通过验收的个人美食地图 + 已核验武汉扫街榜”推进为一个**可长期维护、可真实数据刷新、可被智能体稳定调用的本地优先美食地图工具**。

本阶段不改变产品边界：用户的个人记录仍然存储在浏览器本地；扫街榜仍然是推荐覆盖层，不会自动变成个人记录；任何未核验或存在冲突的 POI 不能成为地图图钉。

### 4A.2 历史阶段用户目标

- 我想在榜单数据发生变化时，看到哪些店新增、下架、改名或迁址，而不是直接污染地图。
- 我想在详情页知道图片、街景或公开证据是否与门店匹配。
- 我想更快录入自己的到访记录，尤其是名称、评分、时间、照片和标签。
- 我想在很多图钉中按商圈、评分、距离、核验状态和个人/推荐来源快速查找。
- 我想让伴随式 Agent 或其他智能体安全地读取、筛选、导出和建议保存地点。
- 我想确认当前版本经过真实数据、截图、自动化测试和 PRD 规格复检。

### 4A.3 历史阶段包含范围

| 模块 | P8-P14 要求 |
| --- | --- |
| 真实数据刷新 | 支持重新刷新高德公开榜单并生成新增、下架、改名、迁址、冲突报告 |
| 冲突治理 | 冲突项默认不上图；高风险 POI 需要人工确认后才能覆盖已核验图钉 |
| 图片与街景证据 | 详情页展示图片来源、匹配状态、失败 fallback；支持证据失效审计 |
| 输入体验升级 | 优化必填优先录入、照片保存前预览、地址/标签快速输入、重复地点提示 |
| 查看筛选升级 | 支持个人/推荐分组、商圈/评分/距离/核验状态筛选、列表与地图联动优化 |
| Agent 调用增强 | 提供更稳定的 Agent Bridge 契约、只读查询、批量导出、结构化错误和审计日志 |
| 性能与包体 | 处理首屏和 bundle 风险；地图、推荐面板、图片尽量懒加载 |
| 产品化验收 | 完成 PRD 复检、真实数据复检、截图证据、最终审计报告和发布说明 |

### 4A.4 历史阶段不包含范围

| 非目标 | 说明 |
| --- | --- |
| 不做后台数据服务 | 刷新脚本和本地数据仍以纯前端/本地项目方式维护 |
| 不自动覆盖高风险 POI | 数据刷新发现冲突时必须停在审计状态 |
| 不让 Agent 直接绕过 UI/Domain 规则 | Agent 写入必须走同一套校验、POI 准入和导出逻辑 |
| 不追求实时全网榜单 | 本阶段目标是稳定核验和可审计刷新，不是实时排名服务 |

## 4B. 已完成扩展范围：P15

P15 不废弃扫街榜，而是把产品主体验调整为“空个人图起步 + 扫街榜参考图层 + 智能打点 + 标签过滤 + 地图海报传播”。P8-P14 的真实数据核验成果继续保留，扫街榜 50 店作为首页可显隐图层存在，默认隐藏，不自动污染个人地图。

### 4B.1 用户历程

1. 用户第一次进入 `#/map` 时看到空个人图，底图仍是武汉地图。
2. 首页“扫街榜”按钮可显示或隐藏 50 个已核验参考图钉。
3. 用户点击地图后直接打开新增模态窗，并带入点击坐标。
4. 用户可手动录入店名、地址、评分、到访时间、照片、笔记和标签。
5. 用户可粘贴简介板文本或网页链接；本地解析基础候选，外部 Agent 可提交结构化候选。
6. 多候选时，FoodMap 结合坐标距离、当前定位、历史城市/标签和来源置信度排序，并展示理由。
7. 用户确认后才保存图钉，不允许静默自动落图。
8. 用户可打结构化标签：吃没吃过、评价、食物种类、自定义标签。
9. 用户可按标签组合过滤个人图钉。
10. 用户可导出单张地图海报 PNG，利于朋友圈传播。

### 4B.2 范围与非目标

| 类型 | 内容 |
| --- | --- |
| 包含 | 空个人图、扫街榜图层显隐、点击新增、文本识别、Agent 候选、候选排序、结构化标签、标签过滤、地图海报 |
| 非目标 | 任意网页爬虫、账号/云同步、自动保存候选、远程地图瓦片截图导出、默认展示扫街榜 |

## 4C. 已完成扩展范围：P16 真实地点联动闭环

> P16 以“用户真实打点体验是否闭环”为目标，补齐 P15 之后仍存在的关键缺口：真实 POI 多候选召回、当前定位排序、链接/Agent 候选闭环、手机地图 App 跳转、真实地图感海报传播。P16 继续坚持纯前端、本地优先，不引入账号、后端或云同步。

### 4C.1 P16 用户历程

1. 用户进入 `#/map`，看到空个人图、武汉底图和可显隐的扫街榜参考图层。
2. 用户点击地图或点击底部新增，打开新增地点模态窗。
3. 用户输入店名、地址、简介板文本或网页链接。
4. FoodMap 召回多个地点候选；候选可来自本地文本解析、已保存历史、扫街榜参考和外部 Agent 提交的结构化结果。
5. 候选列表显示名称、地址、来源、距离、置信度和排序理由；用户必须手动选择候选，不能静默落图。
6. 若用户允许定位，当前位置只参与候选排序和距离解释；若拒绝定位，则按点击点、历史城市/标签、候选完整度和来源置信度降级排序。
7. 用户确认候选并保存后，地点进入个人图层，支持结构化标签、照片、评分、到访时间和笔记。
8. 用户在详情页点击“打开地图/导航”，可跳转到手机端地图软件或网页地图。
9. 用户按标签、来源、评分、区域等筛选当前个人图钉。
10. 用户导出一张更接近真实地图查看体验的传播图，包含当前筛选结果、地图感背景、图钉和标题说明。

### 4C.2 P16 包含范围

| 模块 | P16 要求 |
| --- | --- |
| 真实地点候选 | 支持同名多门店候选展示；候选至少包含名称、地址、坐标、来源、置信度和理由 |
| 当前定位 | 支持浏览器 Geolocation 授权、拒绝和失败 fallback；定位只用于排序和辅助展示 |
| 候选排序 | 综合当前位置、地图点击点、历史城市/标签、扫街榜参考、Agent 置信度和字段完整度 |
| 链接/Agent 闭环 | 网页链接不内置任意爬虫；Agent 可提交结构化候选，FoodMap 本地排序和确认 |
| 外部地图联动 | 个人地点和可上图推荐地点详情提供外部地图跳转入口 |
| 地图 App 跳转 | 移动端优先生成高德/Apple Maps/通用 geo 或网页地图链接，失败时复制地址/坐标 |
| 真实地图感海报 | 导出传播图应包含当前筛选图钉、地图感背景、标题、标签摘要和生成时间 |
| 移动端体验 | 390x844 下可完成搜索候选、定位排序、保存、打开地图和导出入口操作 |

### 4C.3 P16 非目标

| 非目标 | 说明 |
| --- | --- |
| 不做后端 POI 服务 | 不引入自建地点搜索后端；真实 POI 搜索优先通过前端 provider、公开链接或 Agent 辅助 |
| 不做任意网页爬虫 | 普通网页链接只作为输入线索；复杂解析由外部 Agent 以结构化候选形式回填 |
| 不自动保存候选 | 任何候选必须用户确认后才能成为个人地点 |
| 不强依赖定位授权 | 用户拒绝定位时仍可用点击点、历史记录和文本置信度排序 |
| 不承诺所有地图 App 都能拉起 | 以高德/Apple Maps/通用 URI/网页地图多策略降级，验收以可点击和 fallback 为准 |
| 不改变扫街榜定位 | 扫街榜仍是参考图层，不默认污染个人地图 |

### 4C.4 P16 出门体验标准

- 用户能在移动端从空个人图完成一次“输入店名或链接 -> 多候选选择 -> 保存 -> 打开外部地图 -> 标签过滤 -> 导出传播图”的完整链路。
- 候选排序必须可解释，且定位拒绝场景不能中断流程。
- 外部地图跳转必须出现在地点详情中，而不是隐藏在调试或 Agent 接口里。
- 传播图不能是空白图，不能导出与当前筛选结果明显不一致的地点集合。
- 扫街榜 50 条真实核验基线仍需通过。
- P16 实施细则、候选字段、provider 顺序、排序理由、Agent action 和验收 fixture 以 [P16 Real Place Linking Implementation Contract](./p16-real-place-linking-implementation-contract.md) 为准；如果实施契约无法支撑本节体验目标，必须先修订契约再开发。

## 4D. 已完成扩展范围：P17 体验可信度与移动主路径收束

> P17 以“真实日常使用是否顺手、可信、可纠错”为目标。P16 已经建立真实地点联动闭环，P17 不改变纯前端、本地优先边界，而是收束用户反复反馈的体验问题：待确认地点难处理、详情页信息层级混乱、移动端查看/挪动/跳转不稳定、筛选栏和图钉视觉不够清楚、分享图还不够适合传播、真实数据量下地图缩放仍需性能验收。

### 4D.1 P17 用户历程

1. 用户进入 `#/map`，首页筛选栏只展示高频开关：扫街榜、参考图层、待确认、标签和更多筛选，所有控件在桌面、平板、移动端都完整可见。
2. 用户看到待确认数量后，可进入“待确认工作台”，逐个处理不确定地点。
3. 待确认工作台展示原始输入、当前坐标风险、候选门店列表、证据说明、确认、跳过和手动挪动入口。
4. 用户点击地点详情时，详情页优先展示地点状态、可编辑标签、核心操作和坐标可信度，再展示照片、评分、时间、地址、笔记、候选校准和外部地图入口。
5. 用户可在详情页始终启动“手动挪动图钉”，移动端和窄屏下也能明确进入、确认、取消和保存。
6. 图钉视觉系统稳定区分已核验收藏、待确认收藏、选中态、扫街榜、参考图层和高密度小点状态，点击和缩放时不闪烁。
7. 用户按标签、状态、来源、评分过滤时，地图、列表、详情入口和分享图使用同一筛选状态。
8. 用户导出分享图时，可预览当前视野/当前筛选结果，生成适合朋友圈传播的单张图。
9. 移动端用户可完整完成：打开地图 -> 筛选待确认 -> 进入详情 -> 修改标签 -> 挪动图钉 -> 打开地图 App -> 导出分享图。
10. Agent 或其他智能体可读取待确认队列、候选上下文和当前筛选状态，但不能直接绕过用户确认固化坐标。

### 4D.2 P17 包含范围

| 模块 | P17 要求 |
| --- | --- |
| 待确认工作台 | 聚合待确认/高风险个人地点，支持候选查看、确认、跳过、手动挪动和处理进度 |

## 4G. 已完成扩展范围：P20-C 个人数据治理闭环补齐

> P20 core 已经实现治理工作台、重复合并预览、导入冲突预览、维护历史、Agent 边界和响应式证据，但基于原始 PRD 的复核显示它还不能被表述为“完整满足 P20 治理闭环”。本阶段 P20-C 承接用户认可的开发方向，目标是补齐原始 PRD 中仍未闭合的治理体验：多类型安全批量操作、重复建议忽略/保留/合并、导入冲突策略选择、治理报告导出、stale-reference 分组和完整端到端覆盖。

P20-C 仍不做后端、不做账号、不做云同步、不承诺外部实时 POI 全自动搜索。所有治理能力都必须服务个人地图维护，而不是把 FoodMap 改造成后台管理台。

### 4G.1 P20 用户历程

1. 用户进入 `#/map`，数据健康入口显示当前个人地点的治理状态摘要。
2. 用户打开数据健康中心，能从 pending、high-risk、manual-adjusted、skipped、duplicate-candidate、stale-reference、import-conflict 等分组进入处理。
3. 用户可对多个地点执行至少 3 类安全批量操作：批量加入处理队列、批量标记跳过、批量应用标签、批量标记已复核、批量导出治理报告中的任意三类；所有会改变坐标、验证状态、删除或合并记录的动作仍需逐项确认。
4. 用户看到重复地点建议时，可比较名称、地址、坐标、照片、评分、标签和更新时间，选择保留两条、合并或忽略建议。
5. 用户能查看地点维护历史：候选确认、手动挪动、跳过、标签治理、导入冲突和合并决策。
6. 用户导入 `.foodmap.json` 前能看到冲突预览：新增、更新、重复、跳过、风险项，并能对可处理项选择策略；取消导入或取消策略确认不会改变现有个人图。

| 不自动修复个人数据 | 重复、冲突、坐标风险和导入问题必须先预览，再由用户显式确认 |

### 4M.6 P26 出门体验标准

- Mate70 用户通过固定 URL 进入 FoodMap，能理解当前运行模式、数据边界和失败状态，并完成主要移动路径。
- 发布检查可以复跑，且不会把桌面移动模拟、HDC 调试、临时 URL 或历史截图冒充为固定 URL 真机验收。
- 本地数据维护能力让用户看清问题、选择处理方式并撤销未确认动作；不得出现静默删除、静默合并或静默坐标固化。
- 自动化验收包含 build、unit、真实 scanlist、P18-P25 回归、P26 targeted E2E、静态部署 verifier 和 Mate70 证据。
- PRD、目标架构、里程碑、验收门槛、gap/drawio、E2E matrix、visual checklist 与 P26 final report 对阶段边界和证据口径保持一致。

## 4N. 本阶段目标：P27 中国大陆公网访问闭环

> P27 承接已验收的 P26 移动端发布体验基线。P26 已证明 FoodMap 作为固定 URL 静态 WebApp 可以在 Mate70 上完成主要路径；P27 的目标不是重新做业务功能，也不是做原生鸿蒙应用，而是让中国大陆用户可以通过稳定、公开、HTTPS 的国内静态托管入口访问同一个本地优先 WebApp。

P27 不改变长期产品边界：不做账号、云同步、多人协作、永久公网分享链接、FoodMap 自建后端、HarmonyOS 原生 HAP、AppGallery 上架、离线地图瓦片、外部实时 POI 搜索完成声明或自动数据修复。

当前实现状态：P27 处于文档开发与实现准备阶段。`build:mainland`、`build:edgeone`、`verify:mainland:deployment` 和 EdgeOne deploy adapter 已存在；但如果 EdgeOne 只返回需要登录态或临时 `eo_token` 的受保护预览 URL，该结果只能证明部署包和浏览器 smoke，不可作为中国大陆公网访问出门验收。

### 4N.1 P27 用户目标

1. 我想在中国大陆网络环境中直接打开一个稳定 HTTPS URL 使用 FoodMap，而不是依赖 GitHub Pages、局域网地址、USB/HDC、临时隧道或平台后台登录态。
2. 我想在 Mate70 浏览器中打开该 URL 后继续获得 P26 已验收的地图、详情、筛选、数据包、只读分享 fallback、刷新恢复和本地数据说明体验。
3. 我想明确知道 FoodMap 仍是纯前端本地优先应用：我的个人地点、照片、快照和治理历史留在本机 IndexedDB，跨设备仍靠 `.foodmap.json`。
4. 我想发布者能用可复跑命令证明国内静态托管入口、hash route、manifest/service worker/static assets 和 Mate70 主路径都可用。
5. 我不希望受保护预览、临时 token、平台后台登录态或 GitHub Pages 被误称为中国大陆公网服务。

### 4N.2 P27 产品形态

| 能力 | P27 要求 |
| --- | --- |
| 大陆公网访问入口 | 使用国内静态托管或对象存储/CDN 提供稳定 HTTPS URL，例如 `https://foodmap.example.cn/#/map` |
| 免费优先试验路线 | EdgeOne Pages 作为优先试验路线；如果只得到受保护预览，必须记录为 blocker 而不是 accepted |
| 长期生产路线 | 腾讯云 COS + CDN、阿里云 OSS + CDN 或国内 Nginx 静态托管；自定义域名在大陆服务时需要 ICP 与 HTTPS |
| 构建与部署门禁 | `npm run build:mainland`、`npm run build:edgeone`、`npm run verify:mainland:deployment`、可选 `npm run deploy:edgeone` |
| 真机验收 | Mate70 必须通过稳定公网 URL 打开应用并完成 P26 accepted 主路径 smoke |
| 本地数据边界 | IndexedDB 和 `.foodmap.json` 仍是唯一个人数据持久化和跨设备携带路径 |
| 非目标边界 | 不新增账号、云同步、后端、原生 HAP/AppGallery、离线地图、永久公网分享或实时 POI 完成 |

### 4N.4 P27 验收路径

1. 完成 P27 文档审计、drawio 审核和边界冻结，确认没有云、账号、后端、原生、离线地图、永久分享或实时 POI 过度承诺。
2. 运行 `npm run build:mainland`、`npm run build:edgeone` 和 `npm run verify:mainland:deployment`，证明本地静态产物可部署。
3. 使用 `EDGEONE_DRY_RUN=1 EDGEONE_PROJECT_NAME=foodmap npm run deploy:edgeone` 或等价方式复核部署命令不会在无凭据时误写远端。
4. 若 provider credentials 可用，部署到 EdgeOne/COS/OSS/国内静态 host；若平台只给受保护预览 URL，记录 blocker 并保持 P27 未验收。
5. 对稳定公网 URL 执行 `FOODMAP_MAINLAND_DEPLOY_URL=<stable-url> npm run verify:mainland:deployment`。
6. 使用浏览器自动化或 ChromeCli smoke 验证 `#/map`、刷新、缺失分享 fallback、manifest、service worker 和静态 assets。
7. 在 Mate70 上通过稳定公网 URL 截图或录屏，证明工作台入口、详情/筛选、数据包、只读分享 fallback 或有效 share、刷新恢复和 release status 可用。
8. 执行 PRD 规格检视、P18-P26 回归影响评估、秘密泄漏审计和 P27 final acceptance report。

### 4N.5 P27 非目标

| 非目标 | 说明 |
| --- | --- |

| 不承诺离线地图 | App 壳和失败态可解释网络问题，但地图瓦片离线缓存不是 P27 范围 |

| 不把受保护预览当公网验收 | 带临时 token、后台登录态或访问限制的 URL 只能作为部署包 smoke 证据 |

### 4N.6 P27 出门体验标准

## 6. 全局体验原则

### 6.1 地图第一

首屏就是地图，不做营销落地页。所有面板都浮在地图上，但不遮挡主要地图区域。用户进入后应立刻看到：

- 当前城市或默认地图区域
- 已记录的美食图钉
- 搜索入口
- 图层入口
- 新增地点入口

### 6.2 信息按需出现

V1.0 不在首页展示所有字段。默认首页只展示：

- 地图点位
- 图层数量
- 搜索
- 筛选入口
- 当前选中点位的摘要卡片

完整内容只在详情抽屉或详情页中出现。

### 6.3 编辑动作要轻

新增和编辑都必须能快速完成。必填字段最少化：

- 名称
- 经纬度
- 图层
- 评分
- 到访时间

地址、城市、标签、笔记、照片均可后补。

### 6.4 分享页必须可信

分享页要让用户明确知道：

- 这是本地只读快照。
- 不会修改原始记录。
- 不能公开永久访问。
- 如需跨设备查看，需要导入 `.foodmap.json`。

### 6.5 旅行手账风不等于低效率

视觉可以温暖，但交互必须清晰。按钮、输入框、状态反馈要像工具产品一样直接。

## 11. 用户流程

### 11.1 新增点位：地图点击

```text
进入 #/map
→ 点击地图空白处
→ 直接打开新增地点表单，并带入点击坐标
→ 填写名称、评分、到访时间、图层
→ 可选上传照片、添加标签和笔记
→ 保存
→ 地图出现新图钉
→ 详情抽屉打开
```

### 11.2 新增点位：搜索结果

```text
输入店名/商圈/菜系
→ 展示搜索结果列表
→ 点击结果
→ 地图定位到结果
→ 点击“保存为美食点位”
→ 打开新增表单并预填名称、地址、城市、经纬度
→ 保存
```

### 11.3 查看详情

```text
点击地图图钉
→ 地图聚焦该点
→ 桌面端右侧详情抽屉打开
→ 移动端底部详情抽屉半屏展开
→ 用户查看照片、评分、到访时间、地址、标签、笔记
```

### 11.4 编辑点位

```text
打开点位详情
→ 点击“编辑”
→ 打开编辑表单
→ 修改字段
→ 保存
→ 关闭表单
→ 详情和地图图钉同步更新
```

### 11.5 删除点位

```text
打开点位详情
→ 点击“删除”
→ 出现确认弹窗
→ 确认删除
→ 点位从地图和列表消失
→ 显示 toast：已删除地点
```

### 11.6 筛选地图

```text
点击筛选
→ 选择城市、标签、评分、到访时间
→ 点击“查看结果”
→ 地图只显示符合条件的点位
→ 顶部显示筛选条件数量
→ 点击“清空”恢复全部
```

### 11.7 生成分享快照

```text
点击“分享”
→ 输入快照标题
→ 确认生成
→ 系统生成本地快照
→ 显示分享页入口
→ 用户可打开只读页或导出 .foodmap.json
```

### 11.8 导出导入

```text
点击“导入/导出”
→ 选择导出当前地图或当前快照
→ 下载 .foodmap.json

clean profile 中打开应用
→ 点击导入
→ 选择 .foodmap.json
→ 校验通过
→ 生成本地分享快照
→ 打开只读分享页
```

## 16. UX 验收标准

### 16.1 桌面端

- 首屏可明确识别地图是主角。
- 左侧图层、顶部搜索、右侧详情不会互相遮挡。
- 点位详情打开后，地图仍可见。
- 新增、编辑、删除入口清晰。
- 筛选状态可见，清空筛选入口明确。
- 分享页不出现任何编辑按钮。

### 16.2 移动端

- 地图默认占据主要屏幕空间。
- 搜索栏不遮挡关键图钉。
- 底部抽屉可折叠，不永久遮住地图。
- 图层和筛选入口易触达。
- 新增流程不需要复杂跳转。
- 详情内容可滚动，按钮不重叠。

### 16.3 数据与安全感

- 刷新后数据不丢失。
- 删除前必须确认。
- 导入失败不能破坏当前数据。
- 分享页明确“本地只读”。
- 导出文件名清楚，例如：`foodmap-shanghai-cafe-2026.foodmap.json`。

## 17. V1.0 体验结论

FoodMap V1.0 采用方案四“旅行手账风”后，产品重点应从“功能很多的地图工具”转为“温暖、克制、可持续记录的个人美食地图”。

这一版的成败不取决于装饰是否足够像手账，而取决于三件事：

1. **地图是否始终是主角。**
2. **添加和查看一个美食点位是否足够顺。**
3. **本地分享和导入导出是否让用户觉得清楚、可信、安全。**

V1.0 和后续阶段应先把图层、标签、评分、照片、到访时间、笔记、本地分享、移动端友好体验和本地数据可携带做扎实。复杂自定义字段和推荐算法可以作为后续候选；账号、多人协作、云端同步和公开地图链接不再作为 FoodMap 产品路线目标。
